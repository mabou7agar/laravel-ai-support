## When To Index

Index models when you need:

- semantic search
- follow-up context from previous results
- cross-model contextual answers
- graph-aware retrieval over related entities

## Typical Workflow

```bash
php artisan ai:list-models
php artisan ai:analyze-model "App\\Models\\Invoice"
php artisan ai:vector-index "App\\Models\\Invoice" --with-relationships
php artisan ai:vector-status
```

`ai:analyze-model` is diagnostic only. It does not write to the database, create vectors, or publish graph nodes. It only reads schema metadata, reflects model relationships, and estimates record counts for sizing guidance.

## Test Retrieval

```bash
php artisan ai:vector-search "overdue invoices from last month"
```

## GraphRAG Workflow

For central Neo4j retrieval:

```bash
php artisan ai:neo4j-init
php artisan ai:neo4j-sync --fresh
php artisan ai:neo4j-stats
php artisan ai:neo4j-diagnose
```

If integrity issues are reported:

```bash
php artisan ai:neo4j-repair --apply
```

## Practical Guidance

- start with 1-2 critical models (Invoice, Customer)
- include only relationships needed for answers
- avoid over-indexing noisy text fields
- re-index after schema/meaningful data shape changes

## Preferred Model Contract

Existing vector-content methods:

- `getVectorContent()`
- `getVectorMetadata()`
- `toRAGContent()`

Preferred methods for new work:

- `toSearchDocument()`
- `toGraphObject()`
- `getGraphRelations()`
- `getAccessScope()`
- `toRAGSummary()`
- `toRAGDetail()`
- `toRAGListPreview(?string $locale = null)`

The package now prefers explicit search documents over auto-discovered text.

## Retrieval Result Contract

Graph/vector results standardize on:

- `model_id`
- `model_class`
- `model_type`
- `source_node`
- `app_slug`
- optional `scope_type`, `scope_id`, `scope_label`
- `entity_ref`
- `object`

## If Qdrant Is Used

Run infrastructure health checks:

```bash
php artisan ai:infra-health
```

If `qdrant_self_check` fails, fix connection first before debugging AI behavior.
