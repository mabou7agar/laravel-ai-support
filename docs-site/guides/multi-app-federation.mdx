## Target Topology

- `ai.test` => master orchestrator
- `dash.test` => child app (example: accounting)
- more child apps can be added with the same pattern
- optional central Neo4j => shared read model for cross-app GraphRAG

## Step 1: Child App Setup

In each child app:

```env
AI_ENGINE_NODES_ENABLED=true
AI_ENGINE_IS_MASTER=false
AI_ENGINE_MASTER_URL=https://ai.test
AI_ENGINE_JWT_SECRET=shared_secret
AI_ENGINE_NODE_NAME="Node1"
AI_ENGINE_NODE_SLUG=node-1
AI_ENGINE_NODE_LABEL=client
```

Child app must expose node API routes and pass health checks.

## Step 2: Master App Setup

In master app:

```env
AI_ENGINE_NODES_ENABLED=true
AI_ENGINE_IS_MASTER=true
AI_ENGINE_JWT_SECRET=shared_secret
AI_ENGINE_NODE_NAME="Node2"
AI_ENGINE_NODE_SLUG=node-2
AI_ENGINE_NODE_LABEL=dash
```

If you want central graph reads on the master:

```env
AI_ENGINE_GRAPH_ENABLED=true
AI_ENGINE_GRAPH_BACKEND=neo4j
AI_ENGINE_GRAPH_READS_PREFER_CENTRAL=true
AI_ENGINE_GRAPH_KB_ENABLED=true

AI_ENGINE_NEO4J_URL=http://localhost:7474
AI_ENGINE_NEO4J_DATABASE=neo4j
AI_ENGINE_NEO4J_USERNAME=neo4j
AI_ENGINE_NEO4J_PASSWORD=secret
```

## Local Identity Defaults

- Define node identity in config with `env(...)`, then read it at runtime with `config('ai-engine.nodes.local.*')`.
- If `AI_ENGINE_NODE_NAME` is missing, the config falls back to `APP_NAME`.
- If `AI_ENGINE_NODE_SLUG` is missing, runtime services derive it from the resolved node name.
- If `AI_ENGINE_NODE_LABEL` is missing, it defaults to `master` when `AI_ENGINE_IS_MASTER=true`, otherwise `client`.
- `AI_ENGINE_NODE_ROLE` is still accepted as a backward-compatible alias, but `AI_ENGINE_NODE_LABEL` is preferred.
- The node URL should come from `APP_URL`; runtime services read it through `config('app.url')`.

## Step 3: Create Node Registry File

`config/ai-engine-nodes.json`

```json
{
  "nodes": [
    {
      "name": "InBusiness",
      "slug": "inbusiness",
      "type": "child",
      "url": "https://dash.test",
      "capabilities": ["search", "actions", "rag", "workflows"],
      "status": "active",
      "weight": 1
    }
  ]
}
```

## Step 4: Dry-Run Then Apply

```bash
php artisan ai-engine:nodes-sync --file=config/ai-engine-nodes.json
php artisan ai-engine:nodes-sync --file=config/ai-engine-nodes.json --autofix
php artisan ai-engine:nodes-sync --file=config/ai-engine-nodes.json --apply --prune --ping --force
```

Recommended sequence:

1. dry-run (`nodes-sync` only)
2. optional safe normalization (`--autofix`)
3. apply (`--apply --prune --ping --force`)

## Step 5: Verify Routing

```bash
php artisan ai-engine:test-real-agent --message="list recent records" --json
```

Expected:

- tool usage shows `route_to_node` when ownership is remote
- response envelope stays standard

## Step 6: Cleanup Stale Nodes

```bash
php artisan ai-engine:node-cleanup --status=error --days=0 --apply --force
```

## Operating Pattern for Many Apps

- maintain one central node manifest file
- sync from CI/CD on master
- avoid manual node-by-node edits in production
- use Neo4j as the shared read model when you need cross-app semantic/graph search
- keep node routing for app-owned writes and actions
