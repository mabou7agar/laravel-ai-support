Use `AutonomousModelConfig` classes as the explicit contract between AI and your domain model.

## Example: Invoice Model Config

```php
<?php

namespace App\AI\Configs;

use App\Models\Invoice;
use LaravelAIEngine\Contracts\AutonomousModelConfig;

class InvoiceModelConfig extends AutonomousModelConfig
{
    public static function getModelClass(): string
    {
        return Invoice::class;
    }

    public static function getName(): string
    {
        return 'invoice';
    }

    public static function getDescription(): string
    {
        return 'Customer invoices with status, due dates, and totals.';
    }

    public static function getFilterConfig(): array
    {
        return [
            'user_field' => 'user_id',
            'date_field' => 'invoice_date',
            'status_field' => 'status',
            'amount_field' => 'total_amount',
            'eager_load' => ['customer'],
        ];
    }

    public static function getTools(): array
    {
        return [
            'create_invoice' => [
                'description' => 'Create a new invoice',
                'parameters' => ['customer_id', 'invoice_date', 'items'],
                'requires_confirmation' => true,
                'handler' => function (array $data) {
                    return Invoice::create([
                        'customer_id' => $data['customer_id'],
                        'invoice_date' => $data['invoice_date'],
                        'status' => 'draft',
                        'total_amount' => $data['total_amount'] ?? 0,
                    ]);
                },
            ],
        ];
    }
}
```

## Permissions Per User

Override `getAllowedOperations(?int $userId)` to enforce authorization boundaries.

## Best Practices

- keep tool handlers deterministic
- require confirmation for write/delete operations
- return normalized arrays/models only
- avoid hidden side effects inside handlers

## Tool Handler Context

Model-config tool handlers are called with the selected parameters and the current `UnifiedActionContext`:

```php
use LaravelAIEngine\DTOs\UnifiedActionContext;

'update_invoice_draft' => [
    'description' => 'Patch and validate the current invoice draft',
    'parameters' => [
        'payload_patch' => ['type' => 'array', 'required' => true],
    ],
    'handler' => function (array $params, UnifiedActionContext $context): array {
        return app(\App\Services\AI\InvoiceDraftService::class)->patchAndPrepare(
            sessionId: $context->sessionId,
            userId: (int) $context->userId,
            patch: (array) ($params['payload_patch'] ?? []),
        );
    },
],
```

Use the context for session-scoped drafts, selected entities, user ownership, idempotency metadata, and multi-turn workflows. Existing one-argument handlers remain compatible in PHP, but new write tools should prefer the context-aware signature instead of relying on global auth state.

Handlers may return `metadata.agent_strategy`; the package copies it to the `AgentResponse` strategy and response metadata. This is useful when a host app wants stable response intents such as `business_action_needs_input`, `business_action_prepare`, or `business_action_execute`.

## Business Action Tools

For app-wide write actions that are not owned by a single model config, register deterministic actions in `config/ai-agent.php`:

```php
'tools' => [
    'business_action_catalog' => \LaravelAIEngine\Services\Agent\Tools\BusinessActionCatalogTool::class,
    'prepare_business_action' => \LaravelAIEngine\Services\Agent\Tools\PrepareBusinessActionTool::class,
    'execute_business_action' => \LaravelAIEngine\Services\Agent\Tools\ExecuteBusinessActionTool::class,
    'suggest_business_action' => \LaravelAIEngine\Services\Agent\Tools\SuggestBusinessActionTool::class,
],

'business_actions' => [
    'create_invoice' => [
        'id' => 'create_invoice',
        'module' => 'sales',
        'label' => 'Create invoice',
        'operation' => 'create',
        'required' => ['customer_id', 'invoice_date', 'items'],
        'confirmation_required' => true,
        'prepare' => [\App\Services\AI\InvoiceActionService::class, 'prepareCreate'],
        'handler' => [\App\Services\AI\InvoiceActionService::class, 'create'],
        'suggest' => [\App\Services\AI\InvoiceActionService::class, 'suggest'],
    ],
],
```

The registry reads `ai-agent.business_actions`, the orchestrator validates required fields, `prepare_business_action` returns a draft without writing, and `execute_business_action` calls the handler only after confirmation when required.

For larger apps, publish actions through package contracts instead of keeping one large config file:

```php
'business_action_providers' => [
    \App\AI\Actions\SalesActionProvider::class,
],

'business_action_relation_resolvers' => [
    \App\AI\Actions\BusinessRelationResolver::class,
],
```

Available package contracts:

- `LaravelAIEngine\Contracts\BusinessActionDefinitionProvider`
- `LaravelAIEngine\Contracts\BusinessActionExecutor`
- `LaravelAIEngine\Contracts\BusinessActionRelationResolver`
- `LaravelAIEngine\Contracts\BusinessActionAuditLogger`
- `LaravelAIEngine\Contracts\ConversationMemory`
- `LaravelAIEngine\Contracts\AgentCapabilityProvider`

`BusinessActionDefinitionProvider` returns action definitions, `BusinessActionExecutor` prepares/executes a domain action through app code, and `BusinessActionRelationResolver` resolves or creates related records around prepare/execute. The package also binds `ConversationMemory` to `CacheConversationMemory` by default, so multi-turn flows are not tied directly to Laravel's cache facade.

Action definitions share the same generic schema for CRUD/workflow operations:

- `operation`: `create`, `update`, `delete`, `status`, `convert`, or `custom`
- `risk`: `low`, `medium`, `high`, or `destructive`
- `confirmation_required`: optional; defaults from `risk`
- `required`, `parameters`, `summary_fields`, `executor`, `handler`, `suggest`, and `relation_resolvers`

For duplicate-submit protection, include `_idempotency_key` or `idempotency_key` in confirmed write payloads, or pass `metadata.idempotency_key` in `UnifiedActionContext`. Bind `BusinessActionAuditLogger` to persist prepare/execute audit records; the default logger is a no-op.

## Action Payload Extraction

Use `LaravelAIEngine\Services\Actions\ActionPayloadExtractor` when a user provides action details conversationally instead of as a complete JSON payload. The service is package-owned and schema-driven: it receives an action definition, the latest message, the current draft payload, and recent history, then returns a sanitized payload patch.

```php
use LaravelAIEngine\Services\Actions\ActionPayloadExtractor;

$patch = app(ActionPayloadExtractor::class)->extract(
    action: $action,
    message: $message,
    currentPayload: $draftPayload,
    recentHistory: $history,
    options: [
        'instructions' => 'Split "5 Alpha Laptop and 4 Beta Phone" into separate items.',
    ],
);
```

The package handles prompt construction, JSON decoding, schema-based sanitization, model selection, and failure fallback. The host app should merge the returned patch into its draft and continue using deterministic validation, permission checks, relation resolution, confirmation, and writes.

For full multi-turn intake, use `LaravelAIEngine\Services\Actions\ActionIntakeCoordinator`. It owns the reusable chat flow around conversation-memory-backed intake payloads, draft payloads, payload extraction, prepare callbacks, execute-after-confirm callbacks, and relation-review hooks. Host apps provide callbacks for domain-specific merge behavior, validation/prepare, execute, and relation lookup.

Configure it with:

```env
AI_AGENT_ACTION_PAYLOAD_EXTRACTION_ENABLED=true
AI_AGENT_ACTION_PAYLOAD_EXTRACTION_ENGINE=openai
AI_AGENT_ACTION_PAYLOAD_EXTRACTION_MODEL=gpt-4o
AI_AGENT_ACTION_PAYLOAD_EXTRACTION_MAX_TOKENS=1400
AI_AGENT_ACTION_PAYLOAD_EXTRACTION_TEMPERATURE=0.1
```

## Capability Memory

If users can ask for actions in many forms, publish action, query, and tool capabilities through an `AgentCapabilityProvider`. The package registry can then feed a host-app memory layer such as Qdrant, Redis, or Neo4j.

Use capability memory for:

- semantic routing to write actions such as `create_invoice`
- tool selection before LLM planning
- module/query discovery
- relation-aware hints such as customer by email or product by SKU
- multilingual or domain-specific aliases

The package provides the contract, DTO, and registry. The host app owns the provider and sync command because modules, permissions, and relations are domain-specific.

See `guides/capability-memory` for the full pattern.

## List UX Hooks Per Model

For cleaner list responses, add model methods:

- `toAISummarySource()` for compact summary-cache source
- `toRAGListPreview(?string $locale = null)` for multi-line, scan-friendly cards

These keep response formatting domain-aware without hardcoding entity-specific logic in global services.

See `guides/entity-list-preview-ux` for full patterns.
