# Agent Skills

Skills are a lightweight orchestration layer above existing agent primitives.

A tool is an atomic callable operation. A skill is a user-facing ability that describes when to use tools, actions, required data, target JSON, and confirmation rules.

For application workflows, prefer this public model:

```text
Chat -> skill detection -> declared tools/actions -> confirmation -> execution
```

Use RAG, Qdrant, Neo4j, and capability search to find context and choose the right skill/tool. Do not put business workflow logic inside RAG prompts. Collectors remain available for schema-driven data collection, but new application actions should normally be modeled as skills backed by tools/actions.

## Structure

```php
namespace App\AI\Skills;

use App\AI\Tools\CreateInvoiceTool;
use App\AI\Tools\FindCustomerTool;
use LaravelAIEngine\Services\Agent\Skills\AgentSkill;

class CreateInvoiceSkill extends AgentSkill
{
    public string $id = 'create_invoice';

    public string $name = 'Create Invoice';

    public string $description = 'Create invoices through approved tools.';

    public array $triggers = ['create invoice', 'new invoice'];

    public array $tools = [
        FindCustomerTool::class,
        CreateInvoiceTool::class,
    ];

    public string $finalTool = CreateInvoiceTool::class;

    public function targetJson(): array
    {
        return [
            'customer_id' => null,
            'items' => [],
        ];
    }
}
```

Keep business writes in host application services and repositories. A skill should reference actions and tools that validate, authorize, confirm, and execute the write.

Class-based skills are auto-discovered from `app/AI/Skills` when manifest fallback discovery is enabled. Class-string tools are normalized to their `getName()` values, so the skill stays refactor-friendly while the runtime still works with stable tool names.

## Scaffold a skill

```bash
php artisan ai:make-skill CreateInvoice
php artisan ai:scaffold skill CreateInvoice
```

This creates an `App\AI\Skills\CreateInvoiceSkill` class extending `AgentSkill` and registers it in the agent manifest under `skill_providers`.

Generated skills are disabled by default and marked as review-required.

## Create tools

Use the simple tool template for custom service calls:

```bash
php artisan ai:make-tool LookupCustomer
php artisan ai:tools:test lookup_customer --payload='{"input":"ACME"}'
```

The generated class extends `SimpleAgentTool`, so most app tools only need stable metadata and a `handle()` method:

```php
use LaravelAIEngine\DTOs\UnifiedActionContext;
use LaravelAIEngine\Services\Agent\Tools\SimpleAgentTool;

class LookupCustomerTool extends SimpleAgentTool
{
    public string $name = 'customers.lookup';

    public string $description = 'Find a customer by name, email, or phone.';

    public array $parameters = [
        'query' => ['type' => 'string', 'required' => true],
    ];

    protected function handle(array $parameters, UnifiedActionContext $context): array
    {
        return [
            'message' => 'Customer lookup completed.',
            'data' => app(CustomerService::class)->findForAi($parameters['query'], $context->userId),
        ];
    }
}
```

For common Eloquent-backed tools, generate property-only templates:

```bash
php artisan ai:make-tool FindCustomer --kind=lookup --model="App\\Models\\Customer"
php artisan ai:make-tool UpsertCustomer --kind=upsert --model="App\\Models\\Customer"
```

Lookup tools extend `ModelBackedLookupTool` and define `$model`, `$search`, and `$returns`.
Upsert tools extend `ModelBackedUpsertTool` and define `$model`, `$identity`, `$write`, `$required`, and `$returns`.

When the app already exposes an action, prefer a thin action-backed tool:

```bash
php artisan ai:make-tool CreateInvoice --kind=action --action=invoices.create
```

`ActionBackedTool` reads the action schema from `ActionRegistry` and executes through `ActionOrchestrator`, so relation resolution, validation, confirmation, idempotency, audit logging, and policies stay in the action flow instead of being duplicated in tool code.

## Scan project abilities

```bash
php artisan ai:skills:discover
php artisan ai:skills:discover --json
php artisan ai:skills:discover --write
```

The scanner proposes draft skills from registered actions, tools, autonomous collectors, and model discovery metadata.

Use `--write` to add draft definitions to `app/AI/agent-manifest.php`. Draft skills are disabled unless `--enable` is passed.

## Test a skill match

```bash
php artisan ai:skills:test "create invoice for ACME"
php artisan ai:skills:test "create invoice for ACME" --include-disabled --json
```

The command shows the matched skill, score, trigger, and compiled execution plan. Action-backed skills compile to the existing `update_action_draft` tool so validation, confirmation, and execution stay inside the action flow.

## Runtime Guardrails

`run_skill` uses the configured skill tool planner to extract target JSON and choose the next declared tool. Planner output is validated against a strict JSON shape before execution:

```json
{
  "action": "ask_user|run_tool|final_response",
  "message": "user-facing message",
  "payload_patch": {},
  "tool_name": "declared tool name or null",
  "tool_params": {}
}
```

Strict validation is enabled by default:

```env
AI_AGENT_SKILL_TOOL_STRICT_SCHEMA=true
```

Tool execution still passes through `AgentExecutionPolicyService`, so host applications can block tools through `ai-agent.execution_policy.tool_deny` or require an allow-list through `tool_allow`. Every `run_skill` result includes `metadata.skill_planner_trace` when the planner makes decisions, which lets apps audit why a tool was selected.

## Diagnose the manifest

```bash
php artisan ai:manifest:doctor
php artisan ai:manifest:doctor --json
```

The doctor reports disabled draft skills, non-executable skills, and skills that reference missing actions or tools.
JSON output also includes an `inventory` section with discovered skill, action, and tool names plus `developer_commands` for the usual scaffold, diagnostic, and pricing dry-run commands. Use it when wiring a host app so you can confirm the AI can see the same tools that your code registered.

Recommended development loop:

```bash
php artisan ai:init
php artisan ai:make-tool LookupCustomer
php artisan ai:tools:test lookup_customer --payload='{"input":"ACME"}' --json
php artisan ai:make-skill CreateInvoice
php artisan ai:skills:discover --write
php artisan ai:manifest:doctor --json
php artisan ai:pricing-simulate fal_ai fal-ai/kling-video/o3/standard/image-to-video --parameters='{"image_url":"https://example.test/product.png"}'
```

Use the orchestration validator when skills also link to sub-agents through `metadata.sub_agents`:

```bash
php artisan ai:test-orchestration
```

It validates skill-to-tool and skill-to-sub-agent links, then can run the focused orchestration tests.

## Manifest entries

```php
return [
    'skill_providers' => [
        'create_invoice' => App\AI\Skills\CreateInvoiceSkill::class,
    ],

    'skills' => [
        'create_invoice' => [
            'name' => 'Create Invoice',
            'description' => 'Create invoices through approved actions.',
            'triggers' => ['create invoice', 'new invoice'],
            'required_data' => ['customer_id', 'items'],
            'actions' => ['invoices.create'],
            'requires_confirmation' => true,
            'enabled' => false,
        ],
    ],
];
```

The manifest is optional during local development if fallback discovery is enabled. For production, keep the manifest checked in when you want deterministic registration without scanning.

## Routing

When `ai-agent.skills.expose_as_capabilities` is enabled, skills are exposed as capability documents through the existing capability registry. This lets semantic routing discover complete abilities without exposing every low-level tool directly.

When `ai-agent.skills.prefer_deterministic_matches` is enabled, exact or strong trigger matches are compiled before the LLM router runs. This keeps common business requests predictable while still using the existing tools, actions, and RAG fallback paths.
