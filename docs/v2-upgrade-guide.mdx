# V2 Upgrade Guide

Orchestration v2 is a breaking runtime refactor. New applications should enter through the runtime boundary instead of broad services.

> **Note (post-v2).** The intermediate v2 `RoutingPipeline` / `RoutingStageContract` routing
> brain has since been removed. `AiNativeRuntime` is now the sole router (an agentic
> tool-calling loop); the only pre-AiNative branch is an active federated node session. See
> [ChatFlow Trace](chat-flow.mdx) for the current runtime model. References to routing stages
> below are retained only to map old names to their current replacements.

## Why This Is Breaking

The old flow mixed routing, RAG selection, action execution, remote node routing, and continuation handling in the same orchestration path. V2 separates those responsibilities:

| Old area | V2 replacement |
| --- | --- |
| `AgentOrchestrator::process()` as the public brain | `AgentRuntimeContract::process()` resolved from the container |
| Embedded route branches | the `AiNativeRuntime` tool-calling loop (sole router) |
| In-method execution branches | AiNative tools, gated by `AgentExecutionPolicyService` |
| RAG decisions inside orchestration | the `search_knowledge` tool plus `RAGPipeline` |
| Ad hoc run state | `ai_agent_runs` and `ai_agent_run_steps` |
| Provider tool side effects without unified run links | provider tool runs, approvals, artifacts, and audit events linked to agent run steps |

## Public Entrypoint

Prefer resolving the runtime contract:

```php
use LaravelAIEngine\Contracts\AgentRuntimeContract;

$response = app(AgentRuntimeContract::class)->process(
    message: $message,
    sessionId: $sessionId,
    userId: $userId,
    options: ['agent_runtime' => 'laravel']
);
```

`LaravelAgentProcessor` is an internal Laravel-runtime processor. New package consumers should not resolve it directly.

## Removed Or Renamed Candidates

These names should not be used as new extension points:

| Removed or renamed name | Status | Replacement |
| --- | --- | --- |
| `AgentOrchestrator` | removed | `AgentRuntimeContract`, `LaravelAgentProcessor` internally |
| `RAGService` / `RAGChatService` | removed | `RAGPipelineContract`, `RAGDecisionEngine` |
| `AutonomousRAGAgent` | renamed | `RAGDecisionEngine` |
| `GenericEntityResolver`, `IntelligentCRUDHandler`, `IntelligentPromptGenerator`, `IntentAnalysisService` | removed | action definitions with `relation_resolvers`, generic module relation config, and tool-driven intake |
| `HasAIFeatures`, `HasAIActions`, `HasSimpleAIConfig`, `HasAIConfigBuilder`, `AutoResolvesRelationships`, `ResolvesAIRelationships`, `AIConfigurable`, `EntityFieldConfig` | removed | skills, tools, action definitions, and explicit relation resolvers |
| `IntelligentEntityService`, `DocumentTypeAnalyzer`, `ai:test-intent` | removed | current agent/action/RAG tests and explicit app tools |
| package demo chat/auth routes, `AIChatController`, `AuthController`, bundled chat Blade widgets, and demo JS assets | removed | versioned package APIs, application-owned UI, and app-defined auth |
| stale intelligent/entity prompt resources | removed | current AI-native skill, action, tool, and RAG prompt resources |
| `DataCollector*`, `AutonomousCollector*`, `ActionIntakeCoordinator`, `ActionPayloadExtractor`, action draft tools | removed | structured collection callbacks or AI-native skills with concrete tools/actions |
| `AutonomousModelConfig` | renamed | `ModelToolConfig` |

Move new behavior into the narrower runtime, routing, execution, RAG, provider-tool, credit, or audit services.

## Removed Collector Storage

The package no longer ships or reads the old collector storage surface:

- `data_collector_configs`
- `data_collector_states`
- `ai_nodes.autonomous_collectors`

If an existing host application already ran those migrations, add an app-owned cleanup migration when you are ready to drop the unused tables/column. New collection flows should use `StructuredCollectionDefinition` for callback/event collection or AI-native skills with concrete tools/actions when the chat should execute a write.

## Config Changes

Runtime config lives under `ai-agent.runtime`:

```php
'runtime' => [
    'default' => 'laravel',
    'langgraph' => [
        'enabled' => false,
        'base_url' => null,
        'fallback_to_laravel' => true,
    ],
],
```

Run storage, retention, queue safety, and event streaming live under `ai-agent.run_retention`, `ai-agent.run_safety`, and `ai-agent.event_stream`.

## Command Changes

New v2 runtime commands:

```bash
php artisan ai:runtime-capabilities
php artisan ai:validate-runtime-config
php artisan ai:agent-runs:recover-stuck
php artisan ai:agent-runs:cleanup-expired
php artisan ai:agent-runs:retention-cleanup
php artisan ai:agent-runs:replay-step {step}
```

Scaffolding is still routed through the existing scaffold command where possible:

```bash
php artisan ai:scaffold tool SendInvoice
php artisan ai:scaffold skill BillingQuestion
php artisan ai:scaffold execution-handler SendInvoice
php artisan ai:scaffold runtime MyRuntime
```

Valid scaffold types include `tool`, `skill`, `execution-handler`, `runtime`, `filter`, and
`rag-retriever`. The removed `routing-stage` type has no replacement — add capabilities as
AiNative tools or skills instead.

## Old-To-New Flow

1. Runtime receives the request.
2. An active federated node session continues or is cleared; otherwise AiNative owns the turn.
3. AiNative plans and calls tools in a loop until it returns a final answer.
4. RAG, tools, sub-agents, approvals, and artifacts stay in their own services, gated by policy.
5. Agent run and step records store trace, status, response, audit, and recovery metadata.
6. API/admin consumers inspect `/api/v1/ai/agent-runs`.

## Live Validation

Use the existing test command for package-level smoke checks:

```bash
php artisan ai:test-orchestration --json
```

Provider-backed checks remain opt-in because they can create billed calls:

```bash
AI_ENGINE_RUN_LIVE_TESTS=true php artisan ai:test-orchestration
AI_ENGINE_RUN_NEO4J_LIVE_TESTS=true php artisan ai:test-orchestration
```

Demo fixtures are intentionally separate from package core. Keep demo tables and seeders generic: projects, tasks, customers, invoices, documents, tenant/workspace scope, approval-required tool runs, credit-limited users, and graph-linked records.
