# Security

Laravel AI Engine exposes powerful capabilities: hosted tools, web search, code execution, computer use, callbacks, generated files, vector retrieval, and app actions. Keep these capabilities explicit and scoped.

## Hosted Tools

Treat these tools as high-risk by default:

- computer use
- remote MCP
- code execution
- hosted shell
- apply patch
- provider skills that can call external systems

Use the provider tool approval layer before exposing them to end users. Approvals and audit events are persisted through the provider tool lifecycle tables.

## Realtime Tool Dispatch

The realtime SDK broker (`POST /api/v1/ai/realtime/tools/dispatch`) executes registered
tools and skills requested during a voice/WebRTC session. It enforces the same org
execution-policy deny-list as the agent loop — a tool in `ai-agent.execution_policy.tool_deny`
is refused with a `policy_blocked` result and never executes — and honours per-tool
`requiresConfirmation()` approvals. Treat the dispatch endpoint as privileged:

- It runs whatever tool name the caller supplies (subject to the deny-list and approvals),
  so keep it behind your authenticated API middleware, not exposed to anonymous browsers.
- The `agent_chat` bridge can read/act on a user's conversation memory keyed by
  `user_id`/`session_id`. Those can arrive as model- or client-supplied tool arguments, so
  trusting them lets a caller load another user's conversation. Binding to the authenticated
  request context is opt-in via `ai-engine.realtime.trust_client_identity` (default
  `false`): while it is `false`, argument-supplied `user_id`/`session_id` are ignored and
  only the context the host established for the request is used. Set it `true` only for
  trusted backend-to-backend callers that intentionally relay identity in the arguments.
- LiveKit session descriptors mint a room token from request `metadata.room` /
  `participant_identity`. Constrain those server-side with the opt-in
  `ai-engine.realtime.livekit` settings: `identity_resolver` / `room_resolver` (closure or
  `Class@method`) override the client value with a server-resolved one, and `allowed_rooms`
  refuses any room not on the allow-list. With none configured (default) behaviour is
  unchanged.

```php
'realtime' => [
    // Ignore argument-supplied identity in agent_chat; bind to the request context only.
    'trust_client_identity' => false,

    'livekit' => [
        // Closure / invokable / 'Class@method' returning the authoritative value (or null).
        'identity_resolver' => fn () => (string) auth()->id(),
        'room_resolver' => null,
        // When non-empty, a requested room must be listed or the token mint is refused.
        'allowed_rooms' => [],
    ],
],
```

## Image Inputs (SSRF / LFI)

Media engine drivers accept request-supplied image inputs. Two are guarded:

- **Remote image URLs** (e.g. Stable Diffusion `init_image_url`) are fetched server-side
  only after passing the SSRF guard (scheme allow-list + private/reserved-IP block, no
  redirects), so a caller cannot point them at `localhost`, link-local, or cloud-metadata
  hosts. Governed by `ai-engine.provider_tools.artifacts.block_private_urls` (default on).
- **Local file paths** in image params are **not** read from the server filesystem by
  default (fail-closed) — a request value like `/etc/passwd` is treated as opaque, not a
  path. Opt in with `ai-engine.security.allow_local_image_paths=true`, or scope reads to
  specific directories via `ai-engine.security.image_path_allowlist`.

## Preview HTML

Structured collection previews are package-rendered and escaped. The package does not ask AI to return HTML or JavaScript.

Recommended defaults:

- Use `component` mode when your frontend framework can render the form.
- Use `html` mode only in trusted chat surfaces.
- Keep JavaScript in published external assets.
- Use CSP in production.
- Never execute HTML returned directly by a model.

## Callback URLs

Callbacks are server-to-server integrations. Protect them with app tokens:

```php
->callbackUrl('https://app.test/api/ai/lead-callback', headers: [
    'Authorization' => 'Bearer '.config('services.ai_callback.token'),
])
```

Callbacks should be idempotent. Store the `session_id`, collection name, user id, and final data hash before writing business records.

## Tenant, Workspace, And User Scope

Do not rely on vector metadata alone for authorization. Use SQL/application records as the source of truth.

Recommended pattern:

1. Pass `user_id`, `tenant_id`, and `workspace_id` through request context.
2. Filter retrieval with metadata for recall quality.
3. Rehydrate candidate records through repositories.
4. Enforce ownership in application services before returning or writing data.

## Hosted Artifacts

Remote artifact persistence rejects private/local URLs by default. Keep private URL blocking enabled for user-facing apps:

```php
'provider_tools' => [
    'artifacts' => [
        'persist_remote_files' => true,
        'block_private_urls' => true,
        'max_remote_bytes' => 50 * 1024 * 1024,
        'allowed_extensions' => ['png', 'jpg', 'mp4', 'wav', 'pdf', 'csv', 'json'],
        'allowed_mime_types' => ['image/*', 'video/*', 'audio/*', 'application/pdf', 'text/*'],
    ],
],
```

## Provider Tool Ownership And Webhooks

Provider-tool runs, approvals, and artifacts are scoped to an owner when you configure a
resolver. Scoping is opt-in: with no resolver (default), the API is unscoped and your app
is expected to enforce its own authorization. Once a resolver returns an id, every
run/approval/artifact lookup is constrained to that owner and cross-owner access returns
HTTP 403 (closing IDOR on the provider-tool endpoints).

```php
'provider_tools' => [
    // Closure, invokable class, or 'Class@method' returning the current owner id (or null).
    'owner_resolver' => fn () => auth()->id(),

    'fal' => [
        // When set, the FAL catalog webhook requires this shared secret via the
        // `X-Fal-Webhook-Secret` header (or `?webhook_secret=`). Mismatch => HTTP 401.
        'webhook_secret' => env('AI_ENGINE_FAL_WEBHOOK_SECRET'),
    ],
],
```

Secrets inside stored payloads (`authorization`, `server_url`, MCP headers, etc.) are
redacted in API responses while the stored value stays intact for continuation replay.

## Agent Run Endpoints

The agent-run REST endpoints (`GET /agent-runs`, `/{run}`, `/{run}/trace`, `POST
/{run}/resume`, `/{run}/cancel`) expose the full conversation, tool payloads, and trace of
a run. They are owner-scoped by default, exactly like the SSE stream endpoint: a run that
has a `user_id` is only readable/controllable by that authenticated user, and the list
endpoint cannot be widened past the caller via a client-supplied `user_id` filter.

```php
'event_stream' => [
    'access' => [
        // Default ON (fail-closed). Set false only if the host enforces its own authz.
        'authorize_owned_runs' => env('AI_AGENT_AGENT_RUN_AUTHORIZE_OWNED_RUNS', true),
        // Expose runs that have no user_id (default false).
        'allow_anonymous_runs' => env('AI_AGENT_AGENT_RUN_ALLOW_ANONYMOUS_RUNS', false),
        // Optional callable|class with authorize($run, $authUserId): bool to override.
        'authorizer' => null,
    ],
],
```

## Structured Data Queries

The `data_query` tool fails closed by default: a query that resolves to no tenant/user
scope is blocked rather than returning unscoped rows. Mark a query `public` to opt a
specific dataset out, or relax globally (not recommended for multi-tenant apps):

```php
'data_query' => [
    'require_scope' => env('AI_ENGINE_DATA_QUERY_REQUIRE_SCOPE', true),
],
```

## Graph Access Scope

GraphRAG retrieval enforces `CAN_ACCESS` edges per user. By default an unscoped query (no
`canonical_user_id` and no `user_email_normalized`) returns the whole graph — correct for
single-tenant and system queries, but a leak risk for multi-tenant deployments. Enable
scope enforcement so an unscoped query matches nothing instead:

```php
'graph' => [
    'require_access_scope' => env('AI_ENGINE_GRAPH_REQUIRE_ACCESS_SCOPE', false),
],
```

When enabled, always pass the authenticated `user_id` (or an explicit `access_scope`)
through retrieval so real users still resolve via their `CAN_ACCESS` edges.

## Admin UI

The admin UI (`ai-engine.admin_ui`, disabled by default) exposes cross-user run control,
policy management, and provider-tool approvals. Its access gate is **fail-closed**: name
admins explicitly, or opt into the broad modes.

```php
'admin_ui' => [
    'access' => [
        'allowed_user_ids' => [...],   // explicit admin user ids
        'allowed_emails'   => [...],   // or emails
        'allowed_ips'      => [...],   // or trusted IPs (bastion)
        // An EMPTY allowlist does NOT grant every authenticated user — opt in:
        'allow_any_authenticated' => false,
        // Localhost is NOT implicitly trusted (proxies can forge 127.0.0.1) — opt in:
        'allow_localhost'         => false,
    ],
],
```

Provider-tool approve/reject audit attribution is taken from the authenticated identity
only — a client-supplied `actor_id` is never trusted. The `ai:scaffold` command
backslash-escapes descriptions so a generated (autoloaded) artifact cannot be coerced into
executing injected code.

## Credit Guards

Synchronous media services check credits before provider calls and only deduct after successful provider responses. Async FAL video jobs reserve credits before submission and finalize only after the provider accepts or completes the job.

Use [Pricing and billing](pricing-billing.mdx) to audit rates and prevent accidental free or discounted production usage.
