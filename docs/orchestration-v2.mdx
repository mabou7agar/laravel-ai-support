# Orchestration V2

Orchestration v2 makes agent execution explicit: a runtime receives the message, a routing decision is made, the dispatcher executes that decision, and run data is persisted for inspection, recovery, retention, and audit.

Laravel remains the default runtime. LangGraph is optional and should be enabled only for workflows that need checkpointing, graph debugging, human interrupts, or long-running multi-agent branches.

For the concrete `POST /api/v1/agent/chat` request-to-response path, including DTO conversion, sync/async selection, routing trace metadata, dispatcher execution, and response shaping, see [ChatFlow Trace](chat-flow.mdx).

## AI-Native Default

New chat and skill flows should use the AI-native runtime. The model decides intent, planning, tool use, RAG use, follow-up questions, and continuation from the registered skills, tools, conversation, memory, and retrieved context.

Laravel remains responsible for the parts that should not be left to model text:

- request validation and form/schema requirements
- tenant, workspace, user, and credit scope checks
- write confirmations and approval records
- trusted tool-result authority for relation IDs and foreign keys
- audit metadata, traces, artifacts, and persisted run state

If a confirmed tool returns missing fields plus `suggested_tools`, Laravel keeps the original write approval in runtime state and can execute safe suggested lookup/search tools before asking the model to retry the write. This lets tools stay generic while avoiding dead-end chat turns such as "please route this manually" when the package already knows a lookup tool can resolve the missing relation.

AI-native context intelligence gives the model a compact continuation snapshot on every turn. The runtime normalizes tool results into generic outcomes such as `found`, `created`, `updated`, `deleted`, `not_found`, `needs_input`, and `failed`; tracks an active task frame with pending confirmation and completed writes; and includes that snapshot in the prompt beside the raw runtime state. This lets follow-up messages such as "confirm", "change it", "did you create it?", or "add another one" reason from structured state instead of only transcript text.

When a model asks the user for a value that is commonly resolvable by lookup tools, such as price, cost, rate, amount, or total, the runtime checks the matched skill for unused lookup/search/find tools and asks the model to try those tools first. Configure these terms with `ai-agent.ai_native.lookup_before_ask_terms`.

Confirmed write tools also get duplicate protection. When the same confirming tool is called again with the same normalized payload, the runtime returns an already-completed response instead of executing the write a second time. Host tools should still keep their own database-level idempotency for production workflows, but the runtime now protects the common chat-loop retry path.

Optionally, a read request can be prevented from re-triggering a write. After a write completes, an ambiguous follow-up such as "search for the invoice for Mohamed" can bias the planner into re-running the create flow. Set `AI_AGENT_GUARD_READ_INTENT_WRITES=true` (`ai-agent.ai_native.guard_read_intent_writes`) and the runtime nudges the planner back toward a read/lookup tool when the latest message is a read (search/show/list/count) carrying no write intent and no write is in progress. It fails open — never blocking a write that is genuinely in progress (skill scope, pending tool, active objective) or a message that also asks to create/update. Override the verb lists with `ai-agent.ai_native.read_intent_terms` / `write_intent_terms`.

```env
AI_AGENT_AI_NATIVE_ENABLED=true
AI_AGENT_AI_NATIVE_SKILLS=true
```

Expose concrete application tools, such as `find_customer`, `create_customer`, `find_product`, and `create_invoice`. The old action-draft workflow tools are no longer registered by the package.

## Runtime Contract

Custom runtimes implement `LaravelAIEngine\Contracts\AgentRuntimeContract`.

```php
use LaravelAIEngine\Contracts\AgentRuntimeContract;
use LaravelAIEngine\DTOs\AgentResponse;
use LaravelAIEngine\DTOs\AgentRuntimeCapabilities;

final class MyRuntime implements AgentRuntimeContract
{
    public function name(): string
    {
        return 'my_runtime';
    }

    public function capabilities(): AgentRuntimeCapabilities
    {
        return new AgentRuntimeCapabilities(
            runtime: $this->name(),
            tools: true,
            artifacts: true,
            humanApprovals: true
        );
    }

    public function process(string $message, string $sessionId, mixed $userId, array $options = []): AgentResponse
    {
        return AgentResponse::success('Handled by my runtime.');
    }
}
```

Register the runtime in the container and select it with `agent_runtime` or `ai-agent.runtime.default`.

## Routing Model

> The earlier v2 `RoutingStageContract` / `ai-agent.routing_pipeline` mechanism has been
> removed. `AiNativeRuntime` is now the sole router and decides every turn inside an agentic
> tool-calling loop. There is no ordered stage pipeline to register into.

To make the agent reach for a capability on a "billing question", you no longer write a
routing stage — you give it a **tool** (or skill) and let AiNative call it. For example, a
knowledge lookup is the `search_knowledge` tool, and `force_rag` makes AiNative call it
before answering. Application capabilities are registered as tools in the AiNative tool
registry and gated by `AgentExecutionPolicyService`. See [Agent skills](agent-skills.mdx)
and [Sub-agents](agent-sub-agents.mdx).

The only branch that runs before AiNative is an active federated node session
(`routed_to_node`), which continues the remote session or clears it on a new topic.

## Execution Handlers

Execution handlers receive a routing decision and return an `AgentResponse`. Keep authorization, idempotency, audit, and side effects here.

```php
use LaravelAIEngine\Contracts\ExecutionHandlerContract;
use LaravelAIEngine\DTOs\AgentResponse;
use LaravelAIEngine\DTOs\RoutingDecision;
use LaravelAIEngine\DTOs\UnifiedActionContext;

final class SendInvoiceHandler implements ExecutionHandlerContract
{
    public function handle(RoutingDecision $decision, string $message, UnifiedActionContext $context, array $options = []): AgentResponse
    {
        // Call an application service here, not a controller or raw model query.
        return AgentResponse::success('Invoice queued for delivery.');
    }
}
```

## Tools With Approval Policy

Provider tools, MCP tools, computer-use tools, and code execution tools should go through the provider-tool approval lifecycle when risk is non-trivial.

```php
$approval = app(\LaravelAIEngine\Services\ProviderTools\ProviderToolApprovalService::class)
    ->requestApproval($toolRun, [
        'type' => 'computer_use',
        'risk_level' => 'high',
        'requires_approval' => true,
        'decision' => ['url' => 'https://example.com/admin'],
    ]);
```

The approval record links back to the provider tool run, agent run, and agent run step when those ids are provided.

## LangGraph Runtime

Enable LangGraph only when a sidecar service is deployed.

```env
AI_AGENT_RUNTIME_DEFAULT=langgraph
AI_AGENT_LANGGRAPH_ENABLED=true
AI_AGENT_LANGGRAPH_BASE_URL=https://langgraph.internal
AI_AGENT_LANGGRAPH_API_TOKEN=...
AI_AGENT_LANGGRAPH_SIGNATURE_SECRET=...
```

The sidecar contract is:

- `POST /runs` starts a graph run.
- `GET /runs/{id}` returns status.
- `POST /runs/{id}/resume` resumes after input or approval.
- `POST /runs/{id}/cancel` cancels a run.
- `GET /runs/{id}/events` returns events.
- `GET /health` reports availability.

LangGraph interrupts that require approval are mapped into Laravel approval records. Approving and resuming an agent run calls `/runs/{id}/resume`; cancelling calls `/runs/{id}/cancel`.

## Streaming Events

The native runtime emits stable event names:

- `run.started`, `run.completed`, `run.failed`, `run.cancelled`, `run.expired`
- `rag.started`, `rag.sources_found`, `rag.completed`
- `tool.started`, `tool.progress`, `tool.completed`, `tool.failed`
- `sub_agent.started`, `sub_agent.completed`
- `approval.required`, `approval.resolved`
- `artifact.created`
- `final_response.token_streamed`, `final_response.stream_completed`

If a client cannot consume live events, use the run trace API to read persisted fallback events.

## Durable Chat Entry Point

`POST /api/v1/agent/chat` stays synchronous by default. Use `execution_mode=auto` when the client wants the package to queue only work that needs durability. Use `execution_mode=async` or `async=true` when a chat request should always become a durable agent run:

```bash
curl -X POST "$APP_URL/api/v1/agent/chat" \
  -H "Content-Type: application/json" \
  -d '{
    "message": "Create invoice and generate preview",
    "session_id": "thread-123",
    "user_id": "user-uuid",
    "execution_mode": "auto",
    "actions": true,
    "use_rag": false
  }'
```

When auto resolves to async, the controller creates an `ai_agent_runs` row, dispatches `RunAgentJob`, and returns `status_url`, `trace_url`, `stream_url`, and broadcast metadata. Execution continues through the same runtime, approval, continuation, artifact, and trace services used by manually created runs.

## RAG Pipeline

RAG is reached through the AiNative `search_knowledge` tool and stays in RAG services:

- `RAGQueryAnalyzer` expands or normalizes the user query.
- `RAGCollectionResolver` chooses collections.
- `RAGRetriever` delegates to vector, graph, hybrid, SQL, or provider file-search retrievers.
- `RAGContextBuilder` produces a compact context.
- `RAGPromptBuilder` creates the generation prompt.
- `RAGResponseGenerator` returns an `AgentResponse` with normalized sources and citations.

The `search_knowledge` tool calls the pipeline; the AiNative loop never retrieves records directly.

## Run Statuses

Agent runs use these statuses: `pending`, `running`, `waiting_approval`, `waiting_input`, `completed`, `failed`, `cancelled`, and `expired`.

Use the agent run API for inspection and control:

```bash
curl "$APP_URL/api/v1/ai/agent-runs/$RUN_ID"
curl "$APP_URL/api/v1/ai/agent-runs/$RUN_ID/stream?timeout=30&poll=500"
curl "$APP_URL/api/v1/ai/agent-runs/$RUN_ID/trace"
curl -X POST "$APP_URL/api/v1/ai/agent-runs/$RUN_ID/resume" \
  -H "Content-Type: application/json" \
  -d '{"approval_key":"...","payload":{"approved":true}}'
curl -X POST "$APP_URL/api/v1/ai/agent-runs/$RUN_ID/cancel" \
  -H "Content-Type: application/json" \
  -d '{"reason":"User cancelled"}'
```

## Tenancy And Scope

Run locks, duplicate-message keys, LangGraph payloads, and runtime execution checks use the same scope shape:

```php
[
    'tenant_id' => 'tenant-1',
    'workspace_id' => 'workspace-1',
]
```

The runtime reuses `vector-access-control.php`, `TenantResolverInterface`, and `MultiTenantVectorService` where applicable.

## Budgets And Rate Limits

Use `CreditManager` for run budgets:

```php
$credits = app(\LaravelAIEngine\Services\CreditManager::class);
$credits->startRunBudget($runId, $ownerId, ['credits' => 50, 'tokens' => 100000]);
$credits->recordRunUsage($runId, ['total_tokens' => 1200, 'credits' => 1.2]);
$remaining = $credits->remainingRunBudget($runId);
```

`RateLimitManager` accepts runtime, run, tenant, workspace, tool, and provider scope keys for scoped throttling.

## Runtime Capability Discovery

Use the API or artisan command to inspect capabilities before selecting a runtime:

```bash
curl "$APP_URL/api/v1/ai/agent-runs/capabilities"
php artisan ai:runtime-capabilities
```

## Retention, Recovery, And Replay

Retention settings live under `ai-agent.run_retention`. Use:

```bash
php artisan ai:agent-runs:retention-cleanup
php artisan ai:agent-runs:recover-stuck
php artisan ai:agent-runs:cleanup-expired
php artisan ai:agent-runs:replay-step {step}
```

Replay creates a new step linked to the failed step instead of rewriting history.

## Runtime Config

Laravel is the default runtime. LangGraph remains optional and explicit:

```php
'runtime' => [
    'default' => 'laravel',
    'langgraph' => [
        'enabled' => false,
        'base_url' => null,
        'fallback_to_laravel' => true,
    ],
],
```

## Upgrade Notes

The v2 flow is a breaking simplification. New applications should call the runtime boundary and avoid adding behavior to broad orchestration services.
