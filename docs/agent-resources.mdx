# Agent Resources

Expose an Eloquent model to the agent as ready-made `find_<name>` and `create_<name>`
tools — without hand-writing a tool class per entity. `AiResource` builds config-driven
tools on top of `ModelBackedLookupTool` / `ModelBackedUpsertTool` (find-or-create by an
identity key, with confirmation) and registers them into the agent tool registry.

## In code

```php
use LaravelAIEngine\Services\Agent\Tools\AiResource;

AiResource::for(\App\Models\Customer::class)
    ->search(['name', 'email'])          // columns find_customer searches
    ->writable(['name', 'email', 'company']) // create_customer fields
    ->identity(['email'])                // find-or-create key (updateOrCreate)
    ->required(['name', 'email'])
    ->register();                        // registers find_customer + create_customer
```

Call this from a service provider's `boot()` (or anywhere with the container available).

Options:

- `->returns(['id', 'name', 'email'])` — columns returned by the find tool.
- `->with(['items'])` / `->with(['items' => ['product_name', 'quantity']])` — register a
  `show_<name>` tool that returns one record **with its related rows** (a model's
  children). `find_`/`create_` and `data_query` return scalar columns only, so this is how
  the agent sees an invoice's line items, an order's products, etc. — without it, "show
  this X" omits the children and "list the items on X" falls back to the knowledge base.
- `->detail()` — register `show_<name>` even with no relations (record detail by id).
- `->detailOnly()` — register **only** `show_<name>` (no `find_`/`create_`). Use when the
  agent should display a record with its relations but never search or write through this
  resource — e.g. an invoice whose listing is owned by `data_query` and whose creation is
  owned by a skill. Config: `'detail_only' => true`.
- `->lookupOnly()` — register only `find_<name>` (read-only resources).
- `->name('client')` — override the tool name base (defaults to the snaked model name).
- `->defaults(['source' => 'ai'])` or `->defaults(fn ($ctx) => ['user_id' => $ctx->userId])`
  — server-set columns the model never collects from the user.
- `->scopedBy(fn ($ctx, $params) => ['tenant_id' => $ctx->metadata['tenant_id'] ?? null])`
  — constrain every lookup/upsert to a tenant/owner.
- `->confirmationMessage('Create or update this customer.')`.

## In config (zero code)

Declare resources in `config/ai-agent.php` and they auto-register at boot:

```php
'resources' => [
    'customer' => [
        'model'    => \App\Models\Customer::class,
        'search'   => ['name', 'email'],
        'writable' => ['name', 'email', 'company'],
        'identity' => ['email'],
        'required' => ['name', 'email'],
        // 'returns' => ['id', 'name', 'email'],
        // 'lookup_only' => true,
        // 'defaults' => ['source' => 'ai'],
    ],
    'product' => [
        'model'    => \App\Models\Product::class,
        'search'   => ['name', 'sku'],
        'writable' => ['name', 'price', 'sku'],
        'identity' => ['sku'],
    ],
],
```

Use the array form for static config; use `AiResource` in code when you need closures
(tenant scoping, dynamic defaults).

## How it fits

- The generated `create_<name>` tool is a `ModelBackedUpsertTool`: it `updateOrCreate`s by
  the `identity` key (so it find-or-creates, never duplicates) and `requiresConfirmation()`.
- The generated `find_<name>` tool is a `ModelBackedLookupTool` (LIKE search over `search`).
- Both are normal agent tools, so they are gated by `AgentExecutionPolicyService` and appear
  to the AiNative planner like any other tool.

For multi-step flows (collect → resolve → confirm → final write) compose these resource
tools inside an [Agent Skill](agent-skills.mdx).

## Resolving relations deterministically

A skill `->relation()` declaration *nudges* the planner to look up before creating, but it
does not guarantee a related record exists when your final write tool runs. For a final
tool that must link a related record (e.g. `create_invoice` needs a `customer_id`), resolve
it **deterministically** with the `ResolvesAgentRelations` trait instead of hand-coding it
or relying on the model to call a separate create tool:

```php
use LaravelAIEngine\Services\Agent\Tools\Concerns\ResolvesAgentRelations;

class CreateInvoiceTool extends AgentTool
{
    use ResolvesAgentRelations;

    protected function relations(): array
    {
        return [[
            'field'    => 'customer_id',
            'model'    => \App\Models\Customer::class,
            'identity' => ['email'],                                   // find-or-create key
            'map'      => ['email' => 'customer_email', 'name' => 'customer_name'],
            'create'   => ['name', 'email'],                          // required to create
        ]];
    }

    public function execute(array $params, UnifiedActionContext $context): ActionResult
    {
        $params = $this->resolveRelations($params, $context); // customer_id now guaranteed
        // ... persist the invoice ...
    }
}
```

The resolver runs in PHP (no model round-trip): it looks the customer up by email, creates
them when missing, and treats the **user-provided identity as the source of truth** — if the
payload carries a `customer_id` whose email no longer matches (e.g. a loose name match), it
drops it and re-resolves. Override `relationScope()` for tenant isolation.

### Recommendation

- **Read/write a single model** → declare it as a **resource** (`AiResource` / config). The
  planner gets `find_*`/`create_*` tools and decides when to use them.
- **A final tool that must link related records** → resolve those with
  **`ResolvesAgentRelations`**, so correctness does not depend on the planner remembering to
  call a create tool. This is the reliable home for find-or-create.
- Use a **skill** to orchestrate the multi-turn collection and confirmation around them.

## Scaling the toolset (context)

Every tool's schema is injected into the planner prompt, so a large registry inflates
context and can degrade tool selection. Trim what each turn carries via
`ai-agent.ai_native.tool_selection`:

```php
'tool_selection' => [
    'strategy'    => env('AI_AGENT_TOOL_SELECTION', 'all'),
    'always'      => ['search_knowledge', 'data_query'],  // never trimmed
    'limit'       => 12,                                   // for keyword/semantic
    'disclosure'  => env('AI_AGENT_TOOL_DISCLOSURE', 'full'),
],
```

Pick the cheapest that fits — reach for embeddings last:

| strategy | what it does | when |
| --- | --- | --- |
| `all` (default) | every tool, full schema | small toolset; no change |
| `skill_scoped` | when a skill is active, only its tools + core | the safest reduction — never hides a tool the skill needs |
| `keyword` | top-K tools by lexical overlap + core | many flat tools; deterministic, no embeddings |
| `semantic` | top-K by embedding similarity + core | large flat registry; one embedding call/turn (cached tool vectors); fails open |

`disclosure: 'progressive'` is an orthogonal axis: tools are listed by **name + summary
only**, and a `find_tools` meta-tool returns a tool's full parameter schema on demand —
keeping the base prompt small even with hundreds of tools. Combine with any strategy.

A reduced strategy never starves the planner: if a scope/score would be empty it falls
back to the full set, and `semantic` falls back when embeddings are unavailable. Enabling
`skill_scoped` is the recommended first step; move to `keyword`/`semantic`/`progressive`
only when you genuinely have a large, flat toolset.

## Aggregates & analytics

`data_query` answers **count / list / status-filter** questions. It returns scalar columns
for up to a handful of rows — so analytics questions ("total revenue", "average invoice
value", "the most expensive invoice", "which customer spent the most") are **not** safe to
answer from it: the model would eyeball a 10-row sample and report figures that can be an
order of magnitude wrong.

The `aggregate_data` tool computes the real figure in SQL instead. It is registered
automatically (disable with `ai-engine.data_query.aggregate_enabled => false`) and **fails
closed** unless the model declares which columns may be aggregated:

```php
'data_query' => [
    'models' => [
        'invoice' => [
            'class'          => \App\Models\Invoice::class,
            'aliases'        => ['invoice', 'invoices'],
            'aggregatable'   => ['total', 'subtotal', 'tax'],   // numeric columns it may sum/rank
            'groupable'      => ['customer_name', 'status'],     // dimensions it may group by
            'metric_aliases' => ['revenue' => 'total', 'spend' => 'total'],
        ],
    ],
],
```

It supports `sum`, `avg`, `min`, `max`, `count`, ranked records (`top`/`bottom`), and a
per-group breakdown (`group_by`). The planner fills these structured params from the
question, but they are also inferred from natural language. Examples it now answers exactly:

| Question | operation |
| --- | --- |
| "total revenue across all invoices" | `sum` of `total` |
| "average invoice value" | `avg` of `total` |
| "which invoice is the most expensive" | `top` records by `total` |
| "the cheapest invoice" | `bottom` records by `total` |
| "which customer has spent the most" | `sum` of `total` grouped by `customer_name` |

Security: `metric` and `group_by` must be in the `aggregatable` / `groupable` allowlists
**and** exist on the table, so the model can never aggregate or group by an arbitrary
column. The same access scope (user / workspace / tenant) as `data_query` is applied, and a
non-`public` model with no applicable scope is refused.

### Line-item / "best selling" questions

`aggregate_data` aggregates a model's own columns, so "best selling product" (units sold)
or "which product makes the most revenue" must aggregate the **line-item** rows, not the
catalog. Register the line-item table as its own analytics model and the agent routes there
automatically:

```php
'line_item' => [
    'class'          => \App\Models\InvoiceItem::class,
    'aliases'        => ['line item', 'sold', 'sale', 'sales', 'selling'],
    'aggregatable'   => ['quantity', 'total'],
    'groupable'      => ['product_name'],
    'metric_aliases' => ['sold' => 'quantity', 'units' => 'quantity', 'revenue' => 'total'],
],
```

The resolver scores each entity the question names: owning the referenced **metric** wins
first (so "average product **price**" stays on the catalog, while "products by **revenue**"
and "best **selling** product" go to the line items), then grouping by a named **dimension**
(`product_name`). So "best selling product" → line items, `sum(quantity)` grouped by
`product_name`; "average product price" → the catalog's `price`.

**Still out of scope:** computed metrics across columns (e.g. profit / margin =
`sale_price - purchase_price`) — `aggregate_data` sums existing columns, not expressions.
Add a computed column / database view (registered as another analytics model) when you need
those, and only when the cost data is actually populated.
